# コミットワークフロー

## 目的

リポジトリの変更を特定し、論理的なコミットにまとめます。

## タスクステップ

### ステップ 1: 変更を調査

リポジトリ内の未コミットの変更をすべて特定して分析します。

**アクション**:

- `git status` を実行して、変更・追加・削除されたファイルを確認
- `git diff` を実行して、実際の変更内容を確認
- 目的やコンポーネント別に変更を分類

**アウトプット**:

- 関連する変更をグループ化したリストを作成
- 各グループは論理的でアトミックなコミットを表す
- 各グループで影響を受けるファイルをリスト化

### ステップ 2: コミット作成

変更の論理的なグループごとに個別のコミットを作成します。

**要件**:

- `git add` を使用して、各コミット用のファイルをステージング
- 英語で明確で説明的なコミットメッセージを記述
- コミットメッセージのベストプラクティスに従う:
  - 命令形を使用（例: "Add feature" であり "Added feature" ではない）
  - 最初の行は 50 文字以内に収める
  - 必要に応じて本文で追加のコンテキストを提供
  - 該当する場合は issue 番号を参照

**プロセス**:

1. 関連ファイルを一緒にステージング: `git add <files>`
2. 説明的なメッセージでコミット作成: `git commit -m "message"`
3. 変更の論理的なグループごとに繰り返し

### ステップ 3: レビュー

作成したコミットをレビューします。

**アクション**:

- `git log` を実行してコミット履歴を確認
- 各コミットがアトミックで適切に説明されているか検証

**検証**:

- すべての変更がコミットされたことを確認
- コミットメッセージが英語で明確であることを確認
